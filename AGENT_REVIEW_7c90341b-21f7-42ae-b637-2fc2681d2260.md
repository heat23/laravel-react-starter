## Findings

### CRITICAL: Shell Injection in deploy.sh via Unquoted Variables (Lines 113‚Äì120)

**File:** `deploy.sh`
**Lines:** 113‚Äì120
**Severity:** CRITICAL

```bash
REDIS_HOST=$(grep "^REDIS_HOST=" "$ENV_SOURCE" | cut -d'=' -f2 | sed 's/#.*//' | tr -d '"' | tr -d "'" | xargs || echo "127.0.0.1")
REDIS_PORT=$(grep "^REDIS_PORT=" "$ENV_SOURCE" | cut -d'=' -f2 | sed 's/#.*//' | tr -d '"' | tr -d "'" | xargs || echo "6379")
REDIS_PASSWORD=$(grep "^REDIS_PASSWORD=" "$ENV_SOURCE" | cut -d'=' -f2 | sed 's/#.*//' | tr -d '"' | tr -d "'" | xargs || echo "")

# Build redis-cli command with optional auth
REDIS_CMD="redis-cli -h $REDIS_HOST -p $REDIS_PORT"
if [[ -n "$REDIS_PASSWORD" && "$REDIS_PASSWORD" != "null" ]]; then
    REDIS_CMD="$REDIS_CMD -a $REDIS_PASSWORD"
fi
```

**Attack Vector:**
If `.env` contains a malicious `REDIS_PASSWORD` value like `$(whoami)` or `; rm -rf /`, it will execute because:
1. The parsed variables are **unquoted** in line 118‚Äì120
2. `REDIS_CMD` is then executed (implicitly or via `$()`)
3. The `xargs` call doesn't prevent command substitution in the resulting variable

**Proof of Concept:**
```bash
# In .env:
REDIS_PASSWORD="null); echo 'injected' #"

# Results in:
REDIS_CMD="redis-cli -h 127.0.0.1 -p 6379 -a null); echo 'injected' #"
# Which executes as shell code when called
```

**Mitigation:**
- Quote all variable expansions: `redis-cli -h "$REDIS_HOST" -p "$REDIS_PORT"`
- If `.env` is user-controlled (local dev), this is lower risk. If `.env` is production-deployed, this is critical.
- Consider using `env-parser` or `source`-ing `.env` safely (with `set -o pipefail`).

### HIGH: Logic Bug ‚Äî Missing `pass` Call After Env Var Validation (Line 156)

**File:** `deploy.sh`
**Lines:** 149‚Äì156
**Severity:** HIGH

```bash
if [[ "$DEPLOY_ENV" != "local" ]]; then
    REQUIRED_VARS=("APP_KEY" "APP_URL" "DB_CONNECTION" "DB_DATABASE")
    for var in "${REQUIRED_VARS[@]}"; do
        if ! grep -q "^${var}=" .env; then
            fail "Missing required env var: $var"
        fi
    done
pass "Required environment variables present"  # <-- WRONG INDENTATION
```

**Issue:** The `pass` call at line 156 is **outdented**, so it always executes even if the loop didn't complete. If a required var is missing, the script calls `fail()` (which exits), so the `pass` should never run. However, the misalignment makes the control flow ambiguous. If a future maintainer modifies this, they may assume `pass` is inside the block.

**Correct Structure:**
```bash
if [[ "$DEPLOY_ENV" != "local" ]]; then
    REQUIRED_VARS=("APP_KEY" "APP_URL" "DB_CONNECTION" "DB_DATABASE")
    for var in "${REQUIRED_VARS[@]}"; do
        if ! grep -q "^${var}=" .env; then
            fail "Missing required env var: $var"
        fi
    done
    pass "Required environment variables present"
else
    warn "Skipping required env var checks for local run"
fi
```

### HIGH: Comparison Operator Mismatch ‚Äî SESSION_ENCRYPT Validation (Line 168)

**File:** `deploy.sh`
**Lines:** 166‚Äì171
**Severity:** HIGH

```bash
SESSION_ENCRYPT_VAL=$(grep '^SESSION_ENCRYPT=' .env 2>/dev/null | cut -d'=' -f2 | tr -d '"' | xargs)
if [[ "$SESSION_ENCRYPT_VAL" != "true" ]]; then
    fail "SESSION_ENCRYPT must be 'true' in production (currently: '${SESSION_ENCRYPT_VAL}'). Set SESSION_ENCRYPT=true in .env before deploying."
fi
```

**Issue:** The validation uses `!=` (string inequality) with `"true"` (string literal). However:
- If `SESSION_ENCRYPT=True` (capital T), this **fails** (case-sensitive comparison)
- If `.env` has `SESSION_ENCRYPT=1` (which Laravel accepts as truthy), this **fails**
- If `.env` is missing `SESSION_ENCRYPT`, the result is empty string, which **fails** (correct, but misleading error message)

**Better Approach:**
```bash
SESSION_ENCRYPT_VAL=$(grep '^SESSION_ENCRYPT=' .env 2>/dev/null | cut -d'=' -f2 | tr -d '"' | xargs || echo "")
SESSION_ENCRYPT_LOWER=$(printf '%s' "$SESSION_ENCRYPT_VAL" | tr '[:upper:]' '[:lower:]')
if [[ "$SESSION_ENCRYPT_LOWER" != "true" && "$SESSION_ENCRYPT_LOWER" != "1" ]]; then
    fail "SESSION_ENCRYPT must be 'true' or '1' in production (currently: '${SESSION_ENCRYPT_VAL:-[unset]}')"
fi
```

This matches Laravel's `env()` behavior, which treats `1` and `true` equivalently.

### MEDIUM: HSTS Preload Configuration ‚Äî User Education Gap (Lines 10‚Äì13 config/security.php)

**File:** `config/security.php`
**Lines:** 10‚Äì13
**Severity:** MEDIUM

```php
// HSTS preload opts the domain into browser preload lists (https://hstspreload.org).
// Only enable after verifying ALL subdomains support HTTPS ‚Äî this is irreversible
// without a 1-year+ wait for removal from preload lists.
'hsts_preload' => env('HSTS_PRELOAD', false),
```

**Issue (Not a Bug):** The comment is good, but it doesn't mention:
1. **Wildcard subdomains**: If using `*.example.com`, ALL subdomains must support HTTPS (breaks staging, development, etc.)
2. **Removal process**: After removal request, there's a 6-month queue + 1-year list TTL = 18 months before full removal
3. **Testing**: Should test with `HSTS_PRELOAD=true` in staging first
4. **Irreversibility:** No programmatic way to remove; requires manual submission to hstspreload.org

**Mitigation:** Add to README or deployment guide. This is non-critical since the default is `false` and the comment warns of irreversibility.
