## Blocking Issues
**Issue #1: Coupon Loss in Swap** (HIGH)
The coupon code is accepted, validated, but never applied. This breaks the UX promise: if a customer navigates to `/pricing?coupon=SAVE20`, enters a subscription change, and clicks "Switch", the coupon is silently dropped.

**Issue #2: SwapConfirmDialog Missing Coupon** (HIGH)
Dialog must pass `coupon` prop to enable its use in the swap endpoint. Currently, only `price_id` is sent.

Fix both by:
1. Adding `coupon?: string` to `SwapConfirmDialogProps`.
2. Passing `couponCode` from parent Pricing component.
3. Including coupon in the router.post() call in the dialog.
4. Updating `SubscriptionController::swap()` to accept and apply the coupon, or explicitly rejecting it if the Stripe SDK doesn't support coupons on subscription updates.

---

## Recommendations (Non-Blocking)
1. **Coupon validation feedback** — Show user confirmation or error after coupon apply.
2. **Comparison page tests** — Add assertions to prevent `relatedComparisons` drift.
3. **Buy page CTA placeholders** — Add TODO comments to prevent shipping with non-functional links.
4. **GuideCta documentation** — Add usage examples and default prop comments.

---

## Conclusion
The marketing pages, comparison framework, and buy page are well-structured and follow established patterns. However, the coupon flow is **incomplete**: validated on the client and server, but never applied to plan swaps. This is a **critical UX bug** that should be fixed before launch. All other issues are minor and do not block production.
