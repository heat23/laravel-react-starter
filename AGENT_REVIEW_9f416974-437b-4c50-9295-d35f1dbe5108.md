## CRITICAL

**Finding: CSP Report-Only Enforcement Gap**

**File:** `config/security.php` line 6
**Severity:** CRITICAL
**Issue:**
The CSP configuration uses a ternary operator to determine `report_only` mode:
```php
'report_only' => (bool) env('CSP_REPORT_ONLY', in_array(env('APP_ENV', 'production'), ['local', 'testing'], true)),
```

However, the `(bool)` cast introduces a subtle semantic problem. When `CSP_REPORT_ONLY` environment variable is explicitly set (even to `'false'` or `'0'` as a string), the cast treats non-empty strings as truthy:
- `env('CSP_REPORT_ONLY', 'false')` → string `'false'` → `(bool)'false'` → **true** ✗ (unintended)
- `env('CSP_REPORT_ONLY', '0')` → string `'0'` → `(bool)'0'` → **false** ✓ (works by luck)

**Impact:**
In production, if an environment variable is set to the string `'false'`, CSP will remain in report-only mode instead of enforcing. This allows violations to be reported but not blocked, creating a window where injection attacks execute without page breakage.

**Recommendation:**
Use explicit string comparison:
```php
'report_only' => env('CSP_REPORT_ONLY') === 'true'
    || in_array(env('APP_ENV', 'production'), ['local', 'testing'], true),
```

**Test Verification:**
SecurityHeadersTest.php tests the final behavior (enforcing vs. report-only header presence) but does NOT test the edge case of `CSP_REPORT_ONLY='false'` in production.

## HIGH

**Finding 1: Unsafe `</script>` Escape Insufficient for Polyglot Attacks**

**Files:**
- `resources/js/Components/seo/BreadcrumbJsonLd.tsx` line 20
- `resources/js/Components/seo/FaqJsonLd.tsx` line 30

**Severity:** HIGH
**Issue:**
The code uses `schema.replace(/<\/script>/gi, '<\\/script>')` to prevent closing tag injection into JSON-LD. However, this is insufficient against polyglot attacks:

1. **JSON string escape:** An attacker can inject content that exploits the JSON context by escaping the JSON string:
   ```json
   {"name": "Test\"></script><script>alert('xss')</script><script>"}
   ```

2. **Example attack vector:**
   Breadcrumb item with name containing: `Test\"></script><script>alert('xss')</script>`
   After `JSON.stringify()`, the double quotes and angle brackets are escaped, but if user data ever bypasses JSON.stringify() or is combined with unescaped user input, the regex is insufficient.

3. **The regex only targets the literal `</script>` string** — it does NOT prevent attribute-based injection or JSON escape sequences if the JSON context is broken.

**Actual Risk Level:**
MEDIUM in practice because:
1. These components receive props from Inertia (server-controlled via JSON encoding)
2. app.blade.php uses `json_encode()` for FAQ data, which properly escapes
3. No user input flows directly into breadcrumb/FAQ names in current codebase

However, if breadcrumb/FAQ data ever sources from user input or a database without JSON encoding, the regex alone is insufficient.

**Recommendation:**
Document that these components MUST receive JSON-safe data. Add a security test with injection payloads to catch regressions if data sources change:
```typescript
// Test with malicious input
const maliciousItem = { name: 'Test\"></script><script>alert("xss")</script>' };
// Verify component does not render XSS
```

**Finding 2: ConsentController Log Injection Risk**

**File:** `app/Http/Controllers/Api/ConsentController.php` lines 22–28
**Severity:** HIGH
**Issue:**
The controller logs user-supplied data without sanitization:

```php
Log::info('cookie_consent_recorded', [
    'categories' => $validated['categories'],
    'version' => $validated['version'] ?? null,
    'timestamp' => $validated['timestamp'] ?? null,
    'ip' => $request->ip(),
    'user_agent' => $request->userAgent(),
]);
```

While `categories` is validated as array of booleans (safe), the `version` and `timestamp` fields are validated as strings with only length constraints (max 20 and 50 bytes).

**Injection Vector:**
An attacker can supply:
- `version: "1.0\nERROR fake_alert=true"`
- `timestamp: "2026-03-20\u0000null_byte_injection"`

Depending on the logging handler (JSON vs. plaintext), this can:
1. Pollute log files with fake ERROR entries (misleading log analysis)
2. Inject newlines to break log parsing (log aggregation system bypass)
3. Inject null bytes (potential filesystem issues)

**Recommendation:**
Either:
1. Remove `version` and `timestamp` from the log (client-side timestamps are unreliable anyway)
2. Or add strict validation: `'timestamp' => ['sometimes', 'date_format:Y-m-d\TH:i:s.u\Z']` and `'version' => ['sometimes', 'regex:/^[\w\.\-]+$/']`
3. Or use structured logging framework that auto-escapes values

**Finding 3: Missing CSRF Protection on Unauthenticated Consent Endpoint**

**File:** `routes/api.php` line 82 and `resources/js/Components/legal/CookieConsent.tsx` line 44
**Severity:** HIGH
**Issue:**
The `/api/consent` endpoint accepts POST requests from unauthenticated users:

```typescript
// CookieConsent.tsx line 44
fetch('/api/consent', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({...}),
}).catch(() => {});
```

The endpoint is under the `api` middleware group. While it's intentionally unauthenticated (must work for guests), the request does NOT include an `X-CSRF-Token` header.

**CSRF Analysis:**
- Laravel's VerifyCsrfToken middleware expects either a token in the request header or a matching cookie
- Unauthenticated API requests may be exempt from CSRF checking depending on middleware configuration
- If not exempt, the fetch call will fail silently (line 52: `.catch(() => {})`)

**Attack Scenario:**
1. Attacker crafts a malicious page that submits a form to `/api/consent`
2. If CSRF protection is not enforced for unauthenticated endpoints, the request succeeds
3. Attacker can log false consent preferences for analytics/compliance tracking

**Verification Needed:**
Check `bootstrap/app.php` and `app/Http/Middleware/VerifyCsrfToken.php` for whether unauthenticated API requests are excluded from CSRF checks.

**Recommendation:**
Explicitly document the CSRF handling for this endpoint. Either:
1. Add to VerifyCsrfToken exclude list with comment explaining why guest consent is exempt
2. Or add CSRF token requirement (requires token in the fetch body)
3. Or rely on SameSite cookie enforcement (already configured in SessionMiddleware)

Current code's silent failure (catch block) hides the issue from users.

## MEDIUM

**Finding 1: SocialAuthService Transaction Race Condition**

**File:** `app/Services/SocialAuthService.php` lines 44–56
**Severity:** MEDIUM
**Issue:**
The `findOrCreateUser()` method checks for existing users OUTSIDE the transaction, then creates inside:

```php
// Lines 28-30: No transaction, no locking
$socialAccount = SocialAccount::where('provider', $provider)
    ->where('provider_id', $socialUser->getId())
    ->first();

// Line 37: No transaction, no locking
$user = User::where('email', $socialUser->getEmail())->first();

// Line 44: Transaction starts HERE
return DB::transaction(function () use ($socialUser, $provider): User {
```

**Race Condition Scenario:**
1. Thread A checks for email `bob@example.com` — not found
2. Thread B checks for same email — not found
3. Both enter DB::transaction at the same time
4. Both attempt to create users with same email
5. Second insert fails with unique constraint violation
6. SocialAuthController catches exception generically (line 146) and returns misleading error

**Likelihood:** LOW (unique constraint on users.email prevents actual duplicates), but exception handling doesn't distinguish between constraint violations and other failures.

**Recommendation:**
Wrap entire method in transaction OR use firstOrCreate:

```php
public function findOrCreateUser(SocialUser $socialUser, string $provider): User
{
    return DB::transaction(function () use ($socialUser, $provider): User {
        // Check social account first
        $socialAccount = SocialAccount::where('provider', $provider)
            ->where('provider_id', $socialUser->getId())
            ->lockForUpdate()
            ->first();
        
        if ($socialAccount) {
            return $socialAccount->user;
        }

        // Use firstOrCreate for atomicity on email lookup
        $user = User::firstOrCreate(
            ['email' => $socialUser->getEmail()],
            [
                'name' => $socialUser->getName() ?? $socialUser->getNickname() ?? 'User',
                'password' => null,
                'signup_source' => $provider,
                'email_verified_at' => now(),
            ]
        );

        return $user;
    });
}
```

**Finding 2: Weak Timestamp Validation in ConsentController**

**File:** `app/Http/Controllers/Api/ConsentController.php` line 19
**Severity:** MEDIUM
**Issue:**
The `timestamp` field allows any string up to 50 bytes without format validation:

```php
'timestamp' => ['sometimes', 'string', 'max:50'],
```

For compliance auditing, timestamps should be valid ISO 8601:

```php
'timestamp' => ['sometimes', 'date_format:Y-m-d\TH:i:s.u\Z'],
```

**Impact:** Reduces auditability if logs contain garbage timestamps instead of valid datetimes.
