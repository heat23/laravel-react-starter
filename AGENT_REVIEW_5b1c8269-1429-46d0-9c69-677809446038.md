## FINDINGS

### CRITICAL

#### 1. TrialNudgeNotification — Misleading Subject When daysLeft = 0

**File:** `app/Notifications/TrialNudgeNotification.php` (lines 68–74)

**Issue:**
```php
$subject = $daysLeft <= 1
    ? "{$appName}: Your trial ends tomorrow"
    : "{$appName}: Your trial ends in {$daysLeft} days";
```

When `$daysLeft == 0`, the condition `$daysLeft <= 1` is TRUE, so the subject becomes **"Your trial ends tomorrow"** — but the trial **ends TODAY (0 days remaining)**, not tomorrow.

**Root cause:** The `daysLeft()` method (line 50) returns `max(0, ...)`, so it can return 0. The subject logic conflates "1 day left" (trial ends tomorrow) with "0 days left" (trial ends today).

**Impact:**
- Users on their last day of trial are told "ends tomorrow" when it actually ends today
- Reduces urgency perception and increases risk of missed payment
- Incorrect messaging creates confusion in customer support

**Evidence from code:**
```php
private function daysLeft(): int
{
    if (! $this->trialEndsAt) {
        return 7;
    }
    return max(0, (int) now()->diffInDays($this->trialEndsAt, false));  // Can return 0
}
```

**Test coverage gap:** No test case for `daysLeft == 0` (would reveal the bug immediately).

**Recommended fix:**
```php
$subject = match (true) {
    $daysLeft === 0 => "{$appName}: Your trial ends today",
    $daysLeft === 1 => "{$appName}: Your trial ends tomorrow",
    default => "{$appName}: Your trial ends in {$daysLeft} days",
};
```

**Severity:** CRITICAL — Users may miss payment deadlines due to misinterpreted messaging.

### HIGH

#### 2. PaymentFailedNotification — Fragile firstName Extraction

**File:** `app/Notifications/PaymentFailedNotification.php` (lines 35–36)

**Issue:**
```php
$fullName = $notifiable->name ?? '';
$firstName = explode(' ', $fullName)[0] ?: $fullName ?: 'there';
```

The extraction logic is **hard to reason about** and relies on loose PHP comparison semantics. While it works, it's fragile:
- If `explode()` returns an empty string at index [0], the chain `'' ?: $fullName ?: 'there'` relies on evaluating empty strings as falsy
- If the name contains only whitespace (e.g., `"   "`), it would return whitespace instead of the fallback
- Maintenance hazard: future developers may not understand the ternary chain logic

**Comparison to other files:**
- `RefundProcessedNotification.php` (line 40): Uses **identical pattern**
- `IncompletePaymentReminder.php` (line 37): Uses **identical pattern**

All three replicate the same fragile pattern, creating a **maintenance hotspot** if the logic needs to change.

**Test coverage:** No test case for `null` user.name or names containing only whitespace.

**Safer alternative:**
```php
$parts = array_filter(explode(' ', $notifiable->name ?? ''));
$firstName = $parts[0] ?? 'there';
```

**Severity:** HIGH — While it works, it's a maintenance hazard. If anyone modifies the ternary chain in one file, all three will behave inconsistently.

#### 5. RefundProcessedNotification — Hardcoded USD Currency Symbol

**File:** `app/Notifications/RefundProcessedNotification.php` (line 38)

**Issue:**
```php
$formattedAmount = '$'.number_format($this->amountRefunded / 100, 2);
```

The class accepts `$currency` as a constructor parameter (line 17):
```php
public readonly string $currency = 'usd',
```

But the formatted amount **always uses `$` (USD symbol)**, ignoring `$this->currency`. A refund in EUR (€100) displays as `$100`, and a refund in GBP (£50) displays as `$50`.

**Impact:**
- Incorrect currency symbols in production emails
- Confusing for international users (users in Europe, UK, Japan, etc.)
- Data integrity issue: email states wrong currency denomination

**Test coverage gap:** All existing tests use `'currency' => 'usd'`. No test for non-USD currencies (EUR, GBP, JPY, AUD, CAD, etc.).

**Recommended fix:**
```php
$symbols = [
    'usd' => '$',
    'eur' => '€',
    'gbp' => '£',
    'aud' => 'A$',
    'cad' => 'C$',
    'jpy' => '¥',
];
$symbol = $symbols[strtolower($this->currency)] ?? strtoupper($this->currency);
$formattedAmount = $symbol . number_format($this->amountRefunded / 100, 2);
```

Or use a Money/Currency library if the project has one available.

**Severity:** HIGH — Affects data correctness for international users and produces incorrect email content.

### MEDIUM

#### 6. WelcomeSequenceNotification — Inconsistent Greeting Format (Uses Full Name vs. First Name)

**File:** `app/Notifications/WelcomeSequenceNotification.php` (lines 48, 72, 94)

**Issue:**
All three emails use the full name in greetings:
```php
->greeting("Hi {$notifiable->name}!")
```

But the other 3 notification classes extract and use **first name only**:
```php
$firstName = explode(' ', $fullName)[0] ?: $fullName ?: 'there';
->greeting("Hi {$firstName},")
```

**Impact:**
- **Tone inconsistency:** User receives "Hi John Michael Doe!" (formal) vs. "Hi John," (friendly) depending on which notification
- **Personalization mismatch:** Billing notifications feel more personal than onboarding
- Not a bug, but reduces perceived email quality across the sequence

**Evidence:** If a user is named "John Michael Doe":
- Welcome email 1: `"Hi John Michael Doe!"` (less personal, longer)
- Payment failed email: `"Hi John,"` (more personal, friendly)
- Refund email: `"Hi John,"` (consistent with payment failed)

**Recommendation:** Standardize all notifications to use firstName for consistency:
```php
$firstName = explode(' ', $notifiable->name ?? '')[0] ?: 'there';
->greeting("Hi {$firstName},")
```

**Severity:** MEDIUM — UX inconsistency. Not a functional bug, but reduces email quality perception.
