## Summary

Welcome page now includes a social proof section (metrics + testimonials) with semantic HTML (`<figure>`, `<blockquote>`, `<figcaption>`). The TypeScript types and prop defaults are well-structured, and conditional rendering is safe. However, **three critical/high-priority issues require fixes:** (1) `githubStars` prop accepted but never passed from controller (feature incomplete), (2) `userCount` hardcoded to 100 instead of database-driven, (3) `testimonials` hardcoded in controller rather than dynamic/configurable, (4) missing accessibility attributes on decorative dividers and section heading, and (5) potential XSS risk if testimonials become user-editable without sanitization.

## Findings

### CRITICAL

**githubStars Prop Never Passed from Controller — Incomplete Feature**
- **Location:** Welcome.tsx line 65 (prop definition), line 430 (conditional render), WelcomeController.php (missing)
- **Issue:** The component accepts `githubStars?: number` and conditionally renders the GitHub stars metric (lines 430–440) if defined. However, `WelcomeController` never passes this prop to Inertia, so `githubStars` will always be `undefined` and the metric will never render. This indicates either:
  1. The feature is incomplete and not yet wired up
  2. The prop should be removed from the interface and component
- **Current code (Welcome.tsx):**
  ```tsx
  interface WelcomeProps {
    githubStars?: number; // Never passed from controller
  }

  {githubStars !== undefined && (
    // GitHub stars metric — never renders
  )}
  ```
- **Expected:** Either pass `githubStars` from the controller with a real value (e.g., via GitHub API call or configuration), or remove the prop and conditional rendering to avoid dead code.
- **Impact:** Feature incomplete — GitHub stars section appears in TypeScript interface but is unreachable at runtime. Creates confusion about whether the feature is implemented.
- **Recommendation:** Choose one:
  1. **Implement:** Add to `WelcomeController`: `'githubStars' => \GitHub::getStarCount()` or fetch from config/cache
  2. **Remove:** Delete prop, interface field, and conditional rendering to avoid dead code confusion

### HIGH

**Hardcoded userCount (100) — Should Be Database-Driven**
- **Location:** WelcomeController.php, line 17
- **Issue:** `userCount` is hardcoded to a static value `100` instead of reflecting actual system state (e.g., total registered users). This metric is meant to convey social proof ("Developers using it") but displaying a fake number undermines credibility.
- **Current code:**
  ```php
  'userCount' => 100,
  ```
- **Expected:** Query from the database:
  ```php
  'userCount' => User::count(),
  ```
- **Impact:** High — social proof metrics should be truthful. If the landing page launches with userCount: 100 but actual users are 5 or 500, it destroys credibility.
- **Recommendation:** Replace with `User::count()` (add `use App\Models\User;` import at top of controller).

**Testimonials Hardcoded in Controller — Not Configurable**
- **Location:** WelcomeController.php, lines 18–34
- **Issue:** Testimonials are hardcoded directly in the controller. For a production app, these should be:
  1. Database-driven (allowing admin UI to manage testimonials)
  2. Or configuration-based (allowing feature flagging or environment-specific testimonials)
  3. The component already has `DEFAULT_TESTIMONIALS` fallback (lines 36–55 in Welcome.tsx), suggesting the intention was to fetch from somewhere else
- **Current code (controller):** Hardcoded array passed to Inertia
- **Current code (component):** Falls back to `DEFAULT_TESTIMONIALS` if prop is empty
- **Problem:** Mismatch between component intent (supports dynamic testimonials) and controller implementation (always passes the same testimonials). To rotate testimonials or add new ones, code must be edited and redeployed.
- **Expected:** Either:
  1. Query from database: `'testimonials' => Testimonial::published()->get()`
  2. Or use config with fallback: `'testimonials' => config('testimonials.social_proof', DEFAULT_TESTIMONIALS)`
- **Impact:** Medium-High — reduces flexibility for product/marketing teams to manage social proof without engineering involvement.
- **Recommendation:** Add a `Testimonial` model (if not exists) and fetch from database, or use configuration file.

**Missing aria-hidden on Decorative Dividers — Accessibility Violation**
- **Location:** Welcome.tsx, lines 438, 445, 450 (three divider elements between metrics)
- **Issue:** Three visual dividers (`<div className="hidden h-10 w-px bg-border sm:block" />`) are hidden on small screens but visible on desktop. They lack `aria-hidden="true"`, which means screen readers may announce them as content breaks, confusing blind/low-vision users on desktop.
- **Current code:**
  ```tsx
  <div className="hidden h-10 w-px bg-border sm:block" />
  ```
- **WCAG Rule:** Decorative elements should have `aria-hidden="true"` (or `role="presentation"`).
- **Expected:**
  ```tsx
  <div className="hidden h-10 w-px bg-border sm:block" aria-hidden="true" />
  ```
- **Impact:** Medium — violates WCAG 2.1 Level AA. Causes unnecessary noise in screen reader output.
- **Recommendation:** Add `aria-hidden="true"` to all three dividers.

### MEDIUM

**Social Proof Section Lacks Visible Heading — Accessibility + UX Issue**
- **Location:** Welcome.tsx, line 426 (`<section aria-label="Social proof">`)
- **Issue:** The social proof section uses only `aria-label="Social proof"` instead of a visible `<h2>` heading. This creates a heading in the accessibility tree but is invisible to sighted users. All other sections (Features, Before/After, Personas, etc.) have visible `<h2>` tags (lines 377, 481, 526, 553, 615, 666). This is inconsistent and violates the principle that visible and accessible structures should match.
- **Current code:**
  ```tsx
  <section aria-label="Social proof" className="container border-t py-24">
    {/* No <h2> heading here */}
  ```
- **Expected:**
  ```tsx
  <section className="container border-t py-24">
    <h2 className="mb-12 text-center text-3xl font-bold">
      Trusted by developers
    </h2>
    {/* Metrics and testimonials */}
  ```
- **Impact:** Medium — violates accessibility guidelines (WCAG 2.1 requires heading hierarchy match between visual and programmatic) and creates cognitive load for sighted users who don't see a clear section title.
- **Recommendation:** Add a visible `<h2>` heading. The aria-label is redundant if a visible heading exists.

**Testimonial Role Field Not Sanitized — Future XSS Risk**
- **Location:** Welcome.tsx, line 469 (`<p className="text-xs text-muted-foreground">{testimonial.role}</p>`)
- **Issue:** The `testimonial.role` field is rendered directly to JSX without sanitization. Currently, testimonials are hardcoded in the controller (trusted data), so there's no immediate risk. However, if testimonials become user-editable (CMS, admin panel, or user-submitted) in the future without adding sanitization, this becomes an XSS vector.
- **Example of future vulnerability:**
  ```tsx
  role: '<img src=x onerror="alert(1)">Attacker</img>'
  ```
- **Current code:**
  ```tsx
  <p className="text-xs text-muted-foreground">{testimonial.role}</p>
  ```
- **Risk level:** Low today (hardcoded data), High if testimonials become dynamic.
- **Recommendation:** Document the assumption that testimonials are admin-controlled trusted data. If testimonials will be user-submittable later, add DOMPurify sanitization:
  ```tsx
  import DOMPurify from 'dompurify';
  <p className="text-xs text-muted-foreground">
    {DOMPurify.sanitize(testimonial.role, { ALLOWED_TAGS: [] })}
  </p>
  ```

**Testimonials TypeScript Type: Optional But Always Defined at Runtime**
- **Location:** Welcome.tsx, lines 61, 132
- **Issue:** The prop is defined as `testimonials?: Testimonial[]` (optional), but defaults to `DEFAULT_TESTIMONIALS` in destructuring (line 132). This creates a false optional — at runtime, `testimonials` is guaranteed to be an array. TypeScript type should match runtime guarantee.
- **Current code:**
  ```tsx
  interface WelcomeProps {
    testimonials?: Testimonial[];
  }

  const Welcome = ({ testimonials = DEFAULT_TESTIMONIALS, ... }) => {
    // testimonials is NEVER undefined here
  ```
- **Issue:** If someone passes `testimonials={undefined}` via Inertia, it will still default. But if someone reads the TypeScript type, they think it could be undefined and add optional chaining (`testimonials?.length`), which is unnecessary.
- **Recommendation:** Change the type to match runtime guarantee:
  ```tsx
  interface WelcomeProps {
    testimonials: Testimonial[]; // Now required
  }
  ```
  The default is set in the destructuring, so the controller can omit the prop and it will default automatically.
